# SelfHandler — Лог решений

> Зафиксированные продуктовые и архитектурные решения по ходу проектирования. Дополняет [Modules Spec](modules.md) (там детали по модулям) — здесь «почему так» и сквозные договорённости.

## Сквозные архитектурные принципы

- **AI — опциональная надстройка, а не фундамент.** Основа реализуется программно (детерминированно). Приложение полностью работает без LLM. AI усиливает поверх. Для каждой умной фичи есть базовый программный уровень. **TODO:** отдельным этапом пройтись по каждому модулю и продумать его LLM-надстройку.
- **Итоги считает сам модуль** (агрегация рядом с данными). Аналитика — витрина, собирает готовое.
- **Паттерн почти каждой умной фичи:** ручная база → готовые шаблоны/интеграции → LLM сверху.
- **Кандидаты в общие механизмы** (заложить, не дублировать): движок повторяющихся правил (расписание/курсы/события); прогноз запасов и докупки (расходуемые ресурсы); майлстоуны ↔ ступенчатый лимит антипривычек. Сейчас НЕ оверинжинирим — фокус на текущих модулях.

## AI-ассистент (Модуль 11)

- Сквозной слой, видит контекст всех модулей. Сценарии: рекомендации, разбор тренировки, инсайты аналитики, ответы.
- **BYOK (Bring Your Own Key):** провайдер настраивается ЮЗЕРОМ через форму (свой оплаченный API: Claude/OpenAI/кастом). Не в конфиге приложения. Токены платит юзер.
- Учётные данные LLM = данные юзера (провайдер/ключ/модель; несколько с выбором активного). Контракт `LlmProvider` + адаптеры, выбор в рантайме (Laravel Service Container).
- **Безопасность ключей:** шифровать в БД, не отдавать на фронт открыто.
- Референс-провайдер — Claude API; модели/цены сверять, не по памяти.

## Решения по модулям

### Модуль 0 — Профиль
- Антропометрия (вес/рост/возраст/пол/%жира/активность) в профиле — на неё опираются и питание, и тренинг.
- **Журнал замеров** = периодический snapshot (вес, руки, ноги, талия, грудь, ягодицы/таз; расширяемо), запись с датой. Напоминание раз в месяц.
- **Маркер результата:** замыкает цикл цель → питание+тренировки → изменения тела → выводы → корректировка.

### Модуль 2 — Питание
- Основа — приложение друга [calorie-tracker](https://github.com/Podvodila/calorie-tracker) (Vue/TS/Dexie, без бэка). Переиспользуем фронт+модель продукта, хранилище портируем на Laravel+MySQL. Адаптируем сами.
- Приёмы пищи **динамические** (категории или свободные по времени).
- Позиция — блюдо целиком+вес ИЛИ по компонентам+вес (атомарное/составное = рецепт).
- Напитки: вода + чай/кофе/энергетики/кола (с калориями и гидратацией). Норма воды из профиля.
- Норма БЖУ/ккал: Миффлин (дефолт) + Катч-МакАрдл на выбор. Цель = модификатор (профицит/дефицит, с учётом размаха). **Динамический TDEE** — учитывает расход тренировок.
- Оценка рациона = калории + БЖУ + «качество» еды (атрибут продукта).
- Распознавание по фото (будущее): компоненты → БЖУ → калории + полезность.

### Модуль 2а — Добавки и витамины
- Справочник: название, направленность (витамин/спортпит/ноотроп/**анаболик**/др.), форма (вкл. инъекция), доза, объём+цена.
- **Рамка по анаболикам/курсовым:** нейтральный трекер мониторинга (что/когда/сколько/факт/остаток/напоминание). НЕ давать протоколов/схем/дозировок/советов по применению. AI тоже только мониторит.
- Курс + **гибкий движок расписания** (N/день, N/нед по дням, сложные циклы, через день, неделя/пауза).
- Напоминание + повторное если пропустил + факт приёма.
- **Связка запасы→прогноз→финансы:** остаток, прогноз «на сколько хватит», напоминание о докупке, стоимость приёма, подсказка экономии на фасовке.

### Модуль 3 — Тренировки
- Типы (полиморфизм): силовые, кардио (бег/вело/ходьба/плавание), гибкость/йога, спорт.
- Силовая — **2 режима**: простой (констатация) / детальный (подходы×вес×повторы+отдых).
- Справочник упражнений + готовые программы + свои сплиты.
- Прогрессия: история + PR + подсказки. Уровень 1 правила (+2.5кг и т.п.) + Уровень 2 LLM-составитель. Дата следующей → Планнер.
- **Бег** — спец-подтип: уровень (самооценка+корректировка), целевое событие (забег = дата+дистанция), план подготовки 3 уровня (ручной/шаблоны/LLM), метрики (темп/пульс/тип/гео), данные (ручной+интеграции Strava/Garmin/Apple Health+LLM).

### Модуль 4 — Цели
- **Общий механизм с типами** (тело/тренинг/финансы/...). Общие поля + специфика по типу. Несколько активных одновременно. К модулю возвращаемся при Тренировках/Финансах.
- **Майлстоуны** — этапы цели.
- Тип «Тело»: ввод целевой вес+срок ↔ темп/нед, предупреждение о нездоровом темпе.
- Тип «Тренинг»: рабочий вес / кардио-результат / регулярность; прогресс из истории.
- Прогресс разных типов — из разных источников (замеры/веса/баланс).

### Модуль 5 — Планнер
- **Хаб событий всех модулей** + свои задачи + планирование дня (тайм-блоки). Не порождает данные, а планирует/показывает/напоминает.
- Переиспользует **движок расписания** из добавок (кандидат в общий механизм).
- Уведомления: push + в приложении + email/Telegram.
- Внешний календарь опционально (только аппка / + Google/Apple).
- Пропуски: фиксировать пропуск ИЛИ перенос — решает юзер.

### Модуль 6 — Ежедневный отчёт
- **Вечерний ритуал:** итог дня + планирование завтра (через Планнер). + онлайн-сводка дня.
- Сводка из всех модулей. Ручной ввод: самооценка + самочувствие (энергия/стресс/настроение) + дневник.
- **Балл дня** из выполнения. Доставка: напоминание + опц. LLM-саммари.
- Граница с Аналитикой: отчёт = срез одного дня; аналитика = динамика.

### Модуль 7 — Хранилище
- **Гибридная архитектура:** общая база «Элемент» (захват/заголовок/статус/инбокс) + полиморфная специфика по типу (задача/идея/покупка/пункт списка). Паттерн как у Целей/Долгов. Единый быстрый захват + конвертация идея→задача.
- **Задачи:** проекты (основная группировка) + теги (поперечные контексты) + приоритет + статус.
- **Зависимости:** иерархия родитель→дети (самоссылка parent_id, как двухуровневые категории) + флаг «блокер» на ребёнке. Незакрытый блокер не даёт закрыть родителя.
- **Идея:** быстрый ввод → инбокс → сортировка → оформление в проект/цель. Зависимые покупки/задачи как дети (живут вместе с идеей).
- **Покупка (wish-лист):** дочерняя к идее ИЛИ самостоятельная; при покупке → транзакция расхода/рассрочка (Финансы), статус «куплено» разблокирует идею.
- **Списки:** простые пункты + статус/рейтинг; кастомные поля по типу (книга/фильм) — позже.
- **Контейнеры:** Проект (группировка задач/идей, опц. связь с целью) + Список (коллекция пунктов). ❓ общий «контейнер с типом» vs две сущности.
- **Теги** — поперечная классификация (контексты @дом/@работа); кандидат в общий механизм на всё приложение.
- Связи: Финансы (покупка→расход), Планнер (задача с датой→событие), Цели (идея→проект/цель), Отчёт (выполнено за день→балл дня).

### Модуль 8 — Привычки и антипривычки
- Привычки: факт + числовые метрики + время. **Конструктор по «Атомным привычкам»** (habit stacking → рутины; implementation intention → Планнер; правило 2 минут; не разрывать цепь).
- Антипривычки — 2 режима: полный отказ (стрик) ИЛИ **ступенчатый лимит** (энергетики 1/день→5/нед→3/нед, связь с майлстоунами).
- Геймификация: стрики + статистика.

### Модуль 9 — Аналитика
- Ядро: **динамика + корреляции** (главная фишка — выводы об эффективности: сон→энергия, питание→вес и т.д.).
- Периоды: день/нед/мес + произвольный + сравнение периодов.
- Выводы: правила (обязательно) + LLM (опц.).
- **Экспорт** (важно): CSV/PDF (врачу/тренеру, бэкап).

### Модуль 10 — Финансы
- **Много счетов + переводы** (наличка/карты/накопительные/валютные, у каждого баланс; перевод = двойная транзакция, не доход/расход).
- **Мультивалютность сразу** — счета в разных валютах, курсы, пересчёт в базовую для сводной аналитики. Хранить исторический курс на дату перевода.
- **Инвестиции отложены** — портфель активов с котировками позже; накопления = счёт-копилка.
- **Категории двухуровневые** (группа → подкатегория), направление доход/расход не смешиваем, архивирование вместо удаления.
- **Бюджет = лимиты по категориям на месяц** (план/факт, предупреждение о приближении/превышении → канал Планнера).
- **Финансовые цели** (реализуют тип «Финансы» из Модуля 4): накопить N / закрыть кредит, прогресс из баланса накопительного счёта (накопить) или остатка связанного долга (закрыть кредит), майлстоуны, целевая сумма+срок ↔ темп/мес.
- **Долги и обязательства** — общий механизм долга с типами (паттерн как у Целей). Кейсы: рассрочка (N платежей по X/мес), долг человеку, закрыть кредитку.
  - **Оба направления:** «я должен» / «мне должны» (признак направления).
  - **Опциональные %:** ставка/переплата по желанию (рассрочка 0% и долг человеку работают без них); полная амортизация с капанием % — позже.
  - **Два режима графика (флаг):** фиксированный (рассрочка → разворачивается график платежей) / свободное гашение (кредитка/человек → только остаток, платежи когда угодно).
  - Платёж = транзакция со ссылкой на долг, уменьшает остаток. Цель «закрыть кредит» связывается с долгом (прогресс = остаток к нулю). Напоминания/регулярные платежи через общий движок расписания + Планнер.
- **Копилки / накопления на цель** (sinking funds, добавлено 2026-06-13) — «накопить X на ремонт/путешествие». Целевая сумма + накоплено, опц. привязка к категории и сроку.
  - **Где деньги — два режима (выбор):** виртуальная копилка (конверт поверх баланса, не плодим счета) ИЛИ привязка к реальному накопительному счёту.
  - Пополнение разовое или регулярное (через движок расписания). Фин-цель «накопить N» = обёртка со сроком/майлстоунами поверх копилки.
- **Подушка безопасности** (emergency fund, добавлено 2026-06-13) — выделена отдельно: **обязательное** ежемесячное наращивание, **бессрочная** (не разовая цель). Входит в обязательные расходы месяца.
  - **Правило пополнения — три режима (выбор):** фикс-сумма/мес / % от месячного дохода / целевой размер = N месяцев расходов (система считает из средних трат).
  - Просадка → снова «недобрана» → возобновляется наращивание. Концептуально — частный случай копилки с флагами «обязательная+бессрочная».
- **Регулярные операции = доходы И расходы** (симметрия, добавлено 2026-06-13). Движок повторений в обе стороны.
  - **Расходы:** подписки/коммуналка/кредит/аренда.
  - **Доходы:** зарплата (кейс: 3 раза/мес фиксированными суммами, пока грн) → **календарь поступлений** (когда и сколько придёт). Плановый доход месяца = база для бюджета, копилок, «% от дохода» подушки.
  - **Cash flow:** плановый доход − обязательные расходы (регулярные + платежи по долгам + обязательное пополнение подушки) = свободные деньги месяца.
  - Разворачиваются в плановые транзакции, факт получено/оплачено/пропущено.
- **Категории — пример «Медицина → Стоматология»**: регулярная профилактика = плановый расход по подкатегории + напоминание о визите (Планнер). Паттерн «дешёвая профилактика сейчас vs дорогая авария потом» (применим и к ТО авто и т.п.).
- **Итоги считает сам модуль** (балансы, сводный в базовой валюте, факт бюджета, доходы/расходы/сальдо); Аналитика — витрина.
- AI-надстройка опц.: правила (аналитика трат, прогноз обязательных платежей) обязательны; LLM-инсайты/советы по экономии — опционально.

## Ревью-проход и фикс противоречий (2026-06-13)

> Сквозной ревью концепции (4 линзы: пробелы / консистентность / продукт-UX / архитектура). Закрытые противоречия (НЕ переоткрывать):

- **Прогресс фин-цели «накопить N»** = из связанной Копилки (не из «баланса счёта» — копилка может быть виртуальной). Исправлено в Модулях 4 и 10.
- **Связь покупка↔транзакция** определена: полиморфный `TRANSACTION.source` (FK на стороне денег) + инвариант «куплено ⟺ есть транзакция/рассрочка». В ER добавлена PURCHASE.
- **Динамический TDEE:** норма не «плавает» задним числом — считается из **плановой** активности дня; факт уточняет на конец дня. Уровень активности профиля = **базовый/вне спорта** (избегаем двойного учёта тренировок).
- **Ступенчатый лимит антипривычки ≠ майлстоун** (достижение vs ограничение со сменой единицы). Не моделировать одной сущностью вслепую.
- **Единый движок повторений** канонизирован: `RecurringRule` + `PlannedOccurrence`. Раньше назывался по-разному в 2а/5/8/10 — это ОДНО И ТО ЖЕ. Канон — в Modules Spec, раздел Модуля 5. **Полная спека (2026-06-13): [Recurrence Engine](recurrence-engine.md).**
  - Формат правила: **свой набор полей** (freq/interval/by_weekday/by_monthday/times_per_day + `cycle_on`/`cycle_off` для «неделя приёма/пауза») + опц. **`rrule`-строка как запасной выход**.
  - Разворачивание: **материализация с окном вперёд** (+90д) + уникальный `(rule_id, date, slot)` (идемпотентность).
  - Движок хранит статус экземпляра (план/выполнено/пропущено/перенесено); **эскалация и доставка напоминаний — НЕ здесь, а в подсистеме Уведомлений**.
- **Подсистема Уведомлений** спроектирована (2026-06-13): **[Notifications](notifications.md)**. Отделена от Планнера (что vs как) и Движка (статус vs доставка).
  - Каналы: единый контракт (Strategy/Adapter поверх Laravel Notifications), **in-app первым**, push/email/Telegram — адаптеры без переделки (паттерн как BYOK в М11). Закрывает Vision-вопрос «push/telegram/email?».
  - Эскалация: повтор через интервал до отметки (макс K раз), настраиваемо по типу. Читает статус из Движка, живёт в Уведомлениях.
  - Антиспам: тихие часы (ночь) + дневная сводка несрочных. Настройки per-категория/канал.
  - Доставка: Laravel Scheduler+queue, идемпотентность по `(source, escalation_count)`. Уведомление не дублирует доменный статус — при отметке дела авто-закрывается.
- **Сквозные правила схемы** сведены (2026-06-13): **[Data Conventions](data-conventions.md)**.
  - **Деньги:** DECIMAL(19,4) + value object `Money` (amount+currency), не float. Пересчёт валют в момент чтения (текущий/исторический курс).
  - **Полиморфизм — гибрид:** class-table (база+таблицы деталей) для разнополевых типов (Тренировки: силовая/кардио/бег); single-table+nullable/JSON для близких (Цели/Хранилище/Долги/Копилка-Подушка). **Без STI-магии** (`parental` и т.п.) — обычные Eloquent-модели + явные скоупы по типу.
  - **`user_id` на каждой таблице с первого дня** + global scope (multi-user задёшево).
  - **Удаление ≠ архив:** SoftDeletes (тех. корзина) ≠ `is_archived` (доменный, виден в истории/аналитике).
  - **Даты — UTC**, таймзона из профиля. **Единицы — в базовых** (граммы/мл/метры/секунды), конвертация на отображении.
  - **Агрегаты:** кэш-значение + event-пересчёт (Observer) для горячих (баланс/остаток/накоплено) + daily-rollup для аналитики на длинных периодах.
- **Вложения** спроектированы (2026-06-13): **[Attachments](attachments.md)**. Полиморфная `Attachment` + единый `FileStorage`-сервис. Хранение — **локальный диск + disk-абстракция** (Laravel Filesystem, переключение на S3/MinIO сменой драйвера; НЕ BLOB в БД). Файлы приватные (подписанные URL). Потребители: фото еды/тела/чеки/GPX.
- **Внешние интеграции** спроектированы (2026-06-13): **[Integrations](integrations.md)**. **Общий слой** (контракт + адаптеры, как BYOK-LLM/каналы), календари — первый представитель; потом Strava/Garmin (fitness), банк-выписки. `Integration` (OAuth-токены шифрованные) + `SyncedItem` (маппинг локальное↔внешнее для дедупа/конфликтов). Календари — **двусторонняя** синхронизация (экспорт occurrences/событий + импорт внешних как «занятость»). Конфликт на старте — last-write-wins.
- **Профиль = единый источник входов** (антропометрия, **базовая валюта**, таймзона, единицы, тон). Расчёты делают модули. Закрыт open «где базовая валюта» → профиль.
- **Прогноз запаса (2а) ≠ recurring (5/10)** — разные механизмы; докупка добавки = разовый плановый расход.
- **Композитные показатели (балл дня, cash flow)** — осознанное исключение из «итоги считает модуль»: их считает агрегатор (Отчёт/Аналитика) поверх готовых цифр.
- ER open-вопросы частично закрыты с рекомендациями: перевод = две связанные записи; виртуальный конверт = «свободный баланс»; деньги = DECIMAL(19,4)+Money VO; пересчёт валют в момент чтения.
